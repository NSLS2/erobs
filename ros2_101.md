# ROS 2 Training Guide
### Getting Started with Robotics at NSLS-II

This guide is a structured 4-week onboarding path for anyone joining the NSLS-II Robotics Division. By the end, you'll have the foundational skills to understand, simulate, and contribute to the `beamline-robotics-framework` stack.

**Prerequisites:** Linux skills, basic Python/C++ familiarity, comfort with the terminal/bash scripting.

---

## [Week 1] The Foundations 
> *Understanding the communication layer and terminal mastery*

The goal of this week is to get ROS 2 installed and understand **how nodes talk to each other** before writing a single line of code. This communication model is at the heart of everything in this repo.

### Day 1–2: Concepts & Installation

- Read the [ROS 2 Introduction](https://docs.ros.org/en/humble/index.html) — focus on what a **node**, **topic**, **service**, and **action** are
- Install [ROS 2 Humble](https://docs.ros.org/en/humble/Installation/Ubuntu-Install-Debs.html) on Ubuntu 22.04 (Humble is the current LTS used in this repo)
- Verify your install: `ros2 --version`

> **Why Humble?** It's the Long-Term Support (LTS) release — stable, well-supported, and what the lab's hardware drivers are built against.

### Day 3: The CLI

Practice the [Beginner CLI Tools](https://docs.ros.org/en/humble/Tutorials/Beginner-CLI-Tools.html) tutorials. By end of day, you should be able to:

```bash
ros2 topic list                        # list all active topics
ros2 topic echo /some_topic            # print live data from a topic
ros2 service list                      # list available services
ros2 service call /some_service ...    # call a service manually
ros2 node info /some_node              # inspect a running node
```

These commands are your primary debugging tools. You'll use them constantly.

### Day 4–5: The Workspace

Learn to [create a `colcon` workspace](https://docs.ros.org/en/humble/Tutorials/Beginner-Client-Libraries/Creating-A-Workspace/Creating-A-Workspace.html).

```bash
mkdir -p ~/ros2_ws/src
cd ~/ros2_ws
colcon build
source install/setup.bash
```

** Crucial Concept [Underlay vs. Overlay] :**

| Term | What it is | How to source it |
|---|---|---|
| **Underlay** | The system-wide ROS 2 installation | `source /opt/ros/humble/setup.bash` |
| **Overlay** | Your workspace on top of the underlay | `source ~/ros2_ws/install/setup.bash` |

Your overlay inherits everything from the underlay. When you build your own packages, they live in the overlay. Always source the underlay first, then the overlay.

> **Tip:** Add both source lines to your `~/.bashrc` so they apply to every new terminal automatically.

## [Week 2] Development & the API
> *Writing nodes in Python and C++*

Now that you understand how ROS 2 communicates, this week is about writing code that participates in that communication.

### Day 1–2: Python vs. C++

Read the [ROS 2 Client Libraries comparison](https://docs.ros.org/en/humble/Concepts/Basic/About-Client-Libraries.html).

In this repo:
- **Python (`rclpy`)** — used for rapid prototyping, Bluesky integration, and high-level task logic
- **C++ (`rclcpp`)** — used for performance-critical nodes (motion control, real-time loops)

You don't need to master both immediately, but you must be comfortable **reading** both. When reviewing a PR or debugging an issue, you'll encounter either.

### Day 3–5: Publishers, Subscribers & Services

Complete the [Writing a Simple Publisher and Subscriber](https://docs.ros.org/en/humble/Tutorials/Beginner-Client-Libraries/Writing-A-Simple-Py-Publisher-And-Subscriber.html) tutorials (do both the Python and C++ versions).

Then complete the [Writing a Service and Client](https://docs.ros.org/en/humble/Tutorials/Beginner-Client-Libraries/Writing-A-Simple-Py-Service-And-Client.html) tutorial.

**🎯 Week 2 Goal: Create a custom `.msg` file.**

```bash
# In your package's msg/ directory, define a custom message:
# SamplePose.msg
float64 x
float64 y
float64 z
string sample_id
```

Custom messages are how data moves between sensors, the robot arm, and the Bluesky layer in this repo. Understanding how to define and use them is essential. See [`pdf_beamtime_interfaces`](src/pdf/pdf_beamtime_interfaces) for a real example.

## [Week 3] Simulation & Visualization
> *Controlling a virtual arm before touching real hardware*

**Never test untested code on real hardware.** This week is about building confidence in simulation first.

### Day 1–2: RViz & URDF

[RViz](https://github.com/ros2/rviz) is the primary visualization tool. It lets you see the robot's coordinate frames (TFs), planned paths, sensor data, and more.

**Task:**
```bash
# Install the UR description package
sudo apt install ros-humble-ur-description

# Launch RViz with a UR5e model
ros2 launch ur_description view_ur.launch.py ur_type:=ur5e
```

Spend time in RViz understanding:
- **TF frames** — every link of the robot has a coordinate frame; motion planning works in TF space
- **Joint State Publisher** — lets you manually move the arm in RViz
- **URDF** — the XML file that describes the robot's geometry; see [`ur3e_hande_robot_description`](src/custom-ur-descriptions/ur3e_hande_robot_description) for a real example with a gripper attached

### Day 3–5: MoveIt 2 in RViz (Simulated Planning)

With RViz already running, this is the right time to get comfortable with motion planning in simulation — before applying it to real hardware in Week 4.

Follow the [MoveIt 2 Quickstart in RViz](https://moveit.picknik.ai/main/doc/tutorials/quickstart_in_rviz/quickstart_in_rviz_tutorial.html) using the simulated Panda arm first, then try loading the UR configuration from [`ur3e_hande_moveit_config`](src/custom-ur-descriptions/ur3e_hande_moveit_config).

Goals:
- Drag the interactive marker to a target pose and hit **Plan**
- Understand what a collision-free trajectory looks like in RViz
- Intentionally place an obstacle in the planning scene and watch the planner route around it

## [ Week 4] MoveIt 2 & Lab Integration
> *Motion planning and UR-specific workflows*

This is where everything comes together. MoveIt 2 is the motion planning layer — it's how the arm moves from Point A to Point B without hitting a table, the detector, or itself.

### Day 1–3: MoveIt 2

Follow the [MoveIt 2 Quickstart in RViz](https://moveit.picknik.ai/main/doc/tutorials/quickstart_in_rviz/quickstart_in_rviz_tutorial.html).

Key concepts to internalize:

| Concept | What it means in practice |
|---|---|
| **Planning Scene** | The virtual model of the robot + obstacles (table, detector, etc.) |
| **SRDF** | Defines move groups (e.g., `arm`, `gripper`) and collision pairs to ignore |
| **Planning Pipeline** | The algorithm that calculates a collision-free path (OMPL, PILZ, etc.) |
| **Execution** | Sending the planned path to the hardware controllers |

See [`ur3e_hande_moveit_config`](src/custom-ur-descriptions/ur3e_hande_moveit_config) for how MoveIt is configured for our specific arm+gripper setup.

### Day 4: UR Academy — Real Hardware Safety

Before ever being near the real arm, watch the [UR Video Tutorials](https://academy.universal-robots.com/).

Focus on:
- **PolyScope interface** — the teach pendant UI for the physical robot
- **Powering on the arm** safely (initialization sequence, brakes)
- **Emergency Stop (E-Stop)** — know where it is and when to use it without hesitation
- **Freedrive mode** — manually guiding the arm by hand

> ⚠️ **Lab Rule:** You do not operate the real arm unsupervised until cleared by a senior team member. When in doubt, E-stop first, ask questions after.

### Day 5: The Bridge — Bluesky Integration

Review [`bluesky_ros`](src/bluesky_ros) in this repo. This package is the key integration layer between:

- **Bluesky / Ophyd** — the experiment orchestration layer used by beamline scientists
- **ROS 2 Actions** — the long-running task interface used to command the robot

The pattern: an **Ophyd device** wraps a **ROS 2 Action Client**, so a scientist can trigger a sample exchange from a Jupyter notebook using the same Bluesky API they already use for detector control.

Also briefly review [rosbridge_suite](https://github.com/RobotWebTools/rosbridge_suite) to understand how ROS 2 topics can be exposed over WebSockets — useful for web dashboards and external monitoring tools.

### *(Optional)* Isaac Sim & Isaac ROS — Supervised

> ⚠️ **This section is done with a senior team member present.** Isaac Sim requires access to a lab GPU workstation and the ROS 2 bridge setup varies per machine. Do not attempt this solo on your first pass.

The NVIDIA ecosystem is central to our long-term perception and simulation stack. There are two distinct pieces:

| Tool | What it is | What we use it for |
|---|---|---|
| **Isaac Sim** | Photorealistic robot simulator (Omniverse-based) | Simulating the arm, testing motion plans, synthetic data generation |
| **Isaac ROS** | GPU-accelerated ROS 2 packages from NVIDIA | Real-time perception, ArUco/AprilTag detection, depth processing |

#### Isaac Sim

Follow the [Isaac Sim Getting Started Guide](https://docs.isaacsim.omniverse.nvidia.com/latest/installation/install_workstation.html) with your supervisor:

```bash
# Pull the Isaac Sim container (run on lab GPU machine)
docker pull nvcr.io/nvidia/isaac-sim:latest
```

- Enable the **ROS 2 Bridge** extension inside Isaac Sim
- Spawn a UR3e asset and confirm `/joint_states` and `/tf` appear via `ros2 topic list`
- Send a joint position command and watch the simulated arm move

#### Isaac ROS

[Isaac ROS](https://github.com/NVIDIA-ISAAC-ROS) is a suite of hardware-accelerated ROS 2 packages. Most relevant to this repo:

```bash
git clone https://github.com/NVIDIA-ISAAC-ROS/isaac_ros_common

# Key packages:
# isaac_ros_apriltag       — GPU-accelerated ArUco/AprilTag detection (feeds aruco_pose/)
# isaac_ros_image_pipeline — accelerated camera preprocessing
# isaac_ros_nitros         — zero-copy transport between ROS 2 and CUDA
```

Walk through `isaac_ros_apriltag` with your supervisor and see how it connects to the [`aruco_pose`](src/aruco_pose) package in this repo.
